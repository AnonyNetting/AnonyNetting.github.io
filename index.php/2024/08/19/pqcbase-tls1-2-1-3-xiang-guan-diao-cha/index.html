    PQC-Base TLS1.2/1.3相关调查

    发布于 2024-08-19  11 次阅读

    调查背景

    上世纪末以来，针对离散对数问题和整数分解问题的高效量子算法相继被提出，近年来量子计算的快速发展使得现有绝大多数公钥密码系统的安全性受到严重威胁。后量子密码学（Post-Quantum Cryptography，PQC）由于其安全性不依赖于上述算法，作为抵御量子计算威胁的两种方法之一，受到了广泛关注。2023年8月，美国国家标准与技术研究院（NIST）发布了首批标准化的后量子算法[1]，其中包括三种数字签名（DS）方案和一种密钥封装机制（KEM），以及等待进一步审查的其他KEM算法。

    传统的基于离散对数问题和整数分解问题的算法以及后量子算法的应用方向都是构建非对称加密。在信息安全领域，特别是在网络安全中，非对称加密是传输层安全（TLS）协议的基础技术，用于当今绝大多数的安全互联网通信中，包括广泛使用的HTTPS。因此，为了确保PQC算法的实际有效性，除了它们的安全性外，还必须确保它们与现有基础设施（如TLS）的兼容性，或者在可接受的范围内调整现有基础设施以适应PQC。

    近年来，研究人员在PQC算法和专用硬件加速器的开发方面取得了显著进展。研究人员提出了在资源消耗低、速度快和面积-时间积最小方面具有优秀性能的DS和KEM的硬件实现。然而，关于如何将PQC算法硬件加速器，甚至PQC算法本身，集成到现有基础设施中的研究仍然不足[2]。例如，在2024年4月16日发布的Chrome 124版本中，由于默认启用了新的量子抗性X25519Kyber768封装机制，在连接到网站、服务器和防火墙时出现了问题，该机制利用Kyber768（一种在2023年8月标准化的KEM算法）量子抗性密钥协商算法用于TLS/1.3和QUIC连接，以保护Chrome TLS流量免受量子密码分析的威胁。经过调查发现，这些错误并非由Google Chrome中的错误引起，而是由于在请求PQC通信时接收ClientHello信息时，Web服务器未能正确实现TLS，无法处理后量子加密请求的ClientHello消息，导致它们拒绝使用Kyber768量子抗性密钥协商算法的连接，而不是在不支持X25519Kyber768时切换到经典加密。最终，Google发布了帮助用户禁用使用Kyber发送握手信息到服务器功能的文档和补丁。这表明，尽管在理论上TLS/1.3协议支持各种基于非对称加密的握手机制，但至少在服务器端，许多服务器尚未为后量子时代做好准备。

    如上所述，尽管PQC硬件加速器取得了相当大的成果，但PQC在TLS等网络安全协议中的应用仍然相对有限。[3] 提出了关于后量子TLS算法的研究。值得注意的是，后量子TLS的最终发布将由特定的专业组织管理，而不是由单一的学术研究决定。然而，类似于自2019年起发展后量子密码学硬件加速器的趋势，在标准开发之前的学术研究也是有意义的。[4] 研究了如何设计或调整TLS以适应智能基础设施中的PQC。[5] 和 [6] 研究了将PQC集成到通用计算设备和特定嵌入式系统中的可行性和效率。[7] 指出当前的工作过于关注如何将算法部署到硬件上，而忽略了如何与现有基础设施集成的问题，并展示了非常罕见的在FPGA上部署包括PQC算法在内的TLS加速器的成果。参考传统密码学时代服务器端TLS硬件加速器的发展，我认为尽管关于后量子TLS硬件加速方法的研究尚未开始，但它将在未来的网络基础设施中发挥重要作用。

    SSL/TLS的基础知识

    什么是SSL/TLS？

    SSL/TLS是一个安全通信框架，上面可以承载HTTP协议（可以理解为HTTP + TLS => HTTPS）或者SMTP/POP3协议等。SSL(Secure Socket Layer)安全套接层，是1994年由Netscape公司设计的一套协议，并与1995年发布了3.0版本。TLS(Transport Layer Security)传输层安全是IETF在SSL3.0基础上设计的协议，实际上相当于SSL的后续版本。出于这样的历史原因，实际上TLS和SSL意义上基本是等价的，只是在搭配不同的词汇时有特定的习惯用法。
    TLS 的主要用例是对 web 应用程序和服务器之间的通信（例如，web 浏览器加载网站）进行加密。TLS 还可以用于加密其他通信，如电子邮件、消息传递和 IP 语音 (VoIP) 等。

    为什么需要TLS？从何处获取TLS服务？

    TLS 加密可以帮助保护 Web 应用程序免受数据泄露和其他攻击。如今，受 TLS 保护的 HTTPS 是网站的标准做法。

    网站或应用程序要使用 TLS，必须在其源服务器上安装 TLS 证书（由于上述命名混淆，该证书也被称为 SSL 证书）。TLS 证书由证书权威机构颁发给拥有域的个人或企业。该证书包含有关域所有者的重要信息以及服务器的公钥，两者对验证服务器身份都很重要。

    TLS如何工作？

    TLS主要分为两层，底层的是TLS记录协议，主要负责使用对称密码对消息进行加密。
    上层的是TLS握手协议，主要分为握手协议，密码规格变更协议和应用数据协议4个部分。

    • 握手协议负责在客户端和服务器端商定密码算法和共享密钥，包括证书认证，是4个协议中最最复杂的部分。
    • 密码规格变更协议负责向通信对象传达变更密码方式的信号。
    • 警告协议负责在发生错误的时候将错误传达给对方。
    • 应用数据协议负责将TLS承载的应用数据传达给通信对象的协议。

    握手协议:

    当用户端向服务器发起访问时，首先进入如下图所示的握手协议流程（RSA-base TLS1.2为例，与TLS1.3的区别在于TLS1.3整合了Hello和公钥交换；其他还有基于Diffie-Hellman的TLS1.2，下文阐述）

    1. client hello客户端向服务器端发送一个client hello的消息，包含下面内容：
    • 可用版本号
    • 当前时间
    • 客户端随机数
    • 会话ID
    • 可用的密码套件清单
    • 可用的压缩方式清单

    我们之前提到了TLS其实是一套加密框架，其中的有些组件其实是可以替换的，这里可用版本号，可用的密码套件清单，可用的压缩方式清单就是向服务器询问对方支持哪些服务。
    客户端随机数是一个由客户端生成的随机数，用来生成对称密钥，我们将其称作第一随机数。

    1. server hello服务器端收到client hello消息后，会向客户端返回一个server hello消息，包含如下内容：
    • 使用的版本号
    • 当前时间
    • 服务器随机数
    • 会话ID
    • 使用的密码套件
    • 使用的压缩方式

    使用的版本号，使用的密码套件，使用的压缩方式是对步骤1的回答，从1所提供的清单中筛选服务器自身支持的服务，并通知客户端在接下来的会话中使用这些配置。
    服务器随机数是一个由服务器端生成的随机数，用来生成对称密钥，我们将其称作第二随机数。

    1. 可选步骤:certificate
      服务器端发送自己的证书清单，因为证书可能是层级结构的，所以处理服务器自己的证书之外，还需要发送为服务器签名的证书。客户端将会对服务器端的证书进行验证。如果是以匿名的方式通信则不需要证书。
    1. ServerKeyExchange
      这一步骤主要目的是通过特定密钥交换协议完成密钥交换，具体而言，这一步由服务器侧发送由以上步骤已经决定的加密算法的公钥，而服务器侧保留私钥。
    1. 可选步骤:CertificateRequest
      如果是在一个受限访问的环境，比如fabric中，服务器端也需要向客户端索要证书。
      如果并不需要客户端认证，则不需要此步骤。
    2. server hello done
      服务器端发送server hello done的消息告诉客户端自己的消息结束了。
    1. 可选步骤:Certificate
      对步骤5的回应，客户端发送客户端证书给服务器
    2. ClientKeyExchange
      在这个步骤中客户端生成第三个随机数，称作预主密钥（也称预备主密钥）。随后用在步骤4中收到的公钥对这个预主密钥进行加密，将加密结果发送给服务器侧。
      如果是公钥或者RSA模式情况下，客户端将根据客户端生成的随机数和服务器端生成的随机数，生成预备主密码，通过该公钥进行加密，返送给服务器端。
      如果使用的是Diff-Hellman密钥交换协议，则客户端会发送自己这一方要生成Diff-Hellman密钥而需要公开的值。这样服务器端可以根据这个公开值计算出预备主密码。
    1. 可选步骤:CertificateVerify
      客户端向服务器端证明自己是客户端证书的持有者。
    2. ChangeCipherSpec(准备切换密码)
      ChangeCipherSpec是密码规格变更协议的消息，由客户端向服务器侧表示后面的消息将会以前面协商过的密钥进行加密。
    3. finished(握手协议结束)
      客户端告诉服务器端握手协议结束了。
    4. ChangeCipherSpec(准备切换密码)
      服务器端告诉客户端自己要切换密码了。
    5. finished(握手协议结束)
      服务器端告诉客户端，握手协议结束了。
    6. 切换到应用数据协议
      这之后服务器和客户端就是以加密（基于之前步骤所建立的对称加密）的方式进行沟通了。

    基于以上过程，我们整理TLS握手过程，以及过程中服务器侧和用户端各自获得的信息如下。
    对于用户侧，公钥来自Hello确定加密套件后服务器侧生成的成对私钥和公钥中的公钥；第一随机数由自身生成；第二随机数由服务器侧生成，明文传送过来；预主密钥由自身生成；三组随机数共同生成了会话密钥，即对称加密的密钥。
    对于服务器侧，私钥和公钥是Hello确定加密套件后自身生成的成对私钥和公钥；第一随机数由用户侧生成，明文传送过来；第二随机数由自身生成；预主密钥由用户侧生成，由公钥加密后传送过来，并由本地保留的私钥来解密获得；三组随机数共同生成了会话密钥，即对称加密的密钥。
    对于第三者，第三者可以获得的信息包括加密套件，第一，第二随机数，公钥加密后的预主密钥，公钥。很明显，第三者不能获取原本的预主密钥，进而不能得知为接下来对称加密而准备的会话密钥。

    截图来自bilibili 技术蛋老师

    以上是RSA-base的TLS1.2流程。实际上RSA理论上可以被替换成其他任意公钥加密算法，包括我们试图融入的PQC等。

    DH-Base的TLS1.2则略有不同，主要区别在于在DH-Base的密钥交换中，用户侧和服务器侧都有各自的公钥和私钥对。事实上静态的DH由于前向安全问题已经被淘汰，常用的是DHE(E ephemeral)，具体来说流程是：
    1. Client Hello，同样发送TLS版本，加密套件清单，以及第一随机数；
    2. Server Hello，同样确认TLS版本，加密套件，以及第二随机数，同时根据所选定的具体密钥交换算法参数确认所生成的公钥和私钥，将公钥PK-server发送出去；这样实际上包含了上图中的2-5；
    3. 客户端接收到确认的加密套件后也根据具体参数自行生成一组配对的私钥和公钥，将客户端生成的公钥PK-client发送给服务器；同时客户端一侧由于现在保有PK-server和SK-client，可以计算出预主密钥；进一步的，根据第一随机数，第二随机数以及预主密钥，现在客户端可以计算出会话密钥；
    4. 服务器收到PK-client后由于保有PK-client和SK-server，也可以计算出预主密钥；DH算法保证由PK-server和SK-client，PK-client和SK-server各自生成的预主密钥是相同的，也就是说，这样也完成了预主密钥的交换；进一步的，根据第一随机数，第二随机数以及预主密钥，现在服务器可以计算出会话密钥；

    可以看出，DH-Base的密钥交换建立在这样一个数学基础上：f(PK-A, SK-B) = f(PK-B, SK-A). 除了原本的DH算法依赖的离散对数问题外，基于椭圆曲线的问题也有相同的效果，即ECDH(E)算法。

    进一步的，这里进一步简化说明TLS1.3相较于TLS1.2如何作改动。由上述过程可知，在TLS1.2中除后续的验证，确认以外，客户端与服务器之间有效信息的沟通进行了两组来回，这被称作2往返时间(2-RTT)。在TLS1.3中为了压缩RTT，而采用如下图所示的流程。其主要区别在于Client在Client Hello时就将包括了Client侧生成的公钥在内的信息送去了Server，这样Server在收到消息时就集齐了第一，第二随机数，PK-Client, SK-server所有计算会话密钥的信息；同样的，Server返回第二随机数，PK-Server后，客户端也集齐了全部信息。这样成功将2-RTT降低到了1-RTT。对比于TLS1.2我们发现，在TLS1.2中之所以慢1 RTT是因为Client要在收到Server从Client提供清单中的选择结果后才根据所选择的加密套件生成配套的公钥和私钥，那么很明显，TLS1.3之所以能节省1 RTT也是因为TLS1.3所支持的加密套件种类相较于TLS1.2大幅减少，所以Client能够作出Server一定能够接受所提出的加密套件的判断。当然，在其他安全性相关的方面TLS1.3也作出了优化，在此不陈述。

    TLS记录协议

    TLS记录协议主要负责消息的压缩，加密及数据的认证：

    消息首先将会被分段，然后压缩，再计算其消息验证码，然后使用对称密码进行加密。客户端和服务器端都执行这样的加密，以及相对应的解密过程，完成交流。由于是对称密码部分，实际上没有太多可说的地方。

    TLS与PQC的融合

    PQC如何参与TLS？

    根据上一节的内容，我们可以总结出DS和KEM具体来说参与了TLS的以下步骤：

    在TLS1.2的RSA/公钥方案中，KEM参与了用户侧用服务器提供的公钥来加密预主密钥，加密结果送去服务器侧，服务器侧用自身保留的私钥解密预主密钥的过程；PQC-KEM可以完全替代包括RSA在内的公钥密码；
    在TLS1.2的DH方案中似乎不包含PQC算法，因为现有的PQC都不具备如f(PK-A, SK-B) = f(PK-B, SK-A)的特性，因此实际上使用的是Hybrid Post-Quantum。但是由于TLS1.2已经是较多年前发布的，近年来关于Hybrid PQ的研究往往基于TLS1.3。包括Google等在推进的一个算法标准X25519Kyber768即将ECDH(E)和Kyber768基于TLS1.3结合起来，具体的流程如下:(https://medium.com/@hwupathum/x25519kyber768-post-quantum-key-exchange-for-https-communication-70eba681931d)

    1. 客户端生成 X25519 和 Kyber 密钥对，而服务器仅生成 X25519 密钥对。
    2. 客户端将其 X25519 和 Kyber 公钥（在 ClientHello 请求中）传输到服务器。
    3. 在收到 ClientHello 请求后，服务器会生成一个随机密钥（即上文的预主密钥），然后使用客户端的 Kyber 公钥进行封装。然后，服务器将其 X25519 公钥以及封装的预主密钥（在 ServerHello 响应中）发送到客户端。
    4. 客户端在收到 ServerHello 响应后，使用其 X25519 私钥计算 X25519 共享密钥。此外，客户端使用其 Kyber 私钥解封装的预主密钥。
    5. 同时，服务器计算自己的共享密钥。

    可见，Hybrid PQ的基本思路是融合了公钥和DH两种方法。双方都生成DH的公私钥，并在ClientHello时发送客户端公钥，ServerHello时发送服务器公钥，这样双方都集齐了自己的私钥和对方的公钥；仅用户端生成自己的Kyber公私钥，ClientHello时发送公钥给服务器，服务器用Kyber公钥加密随机生成的预主密钥，在ServerHello时返回给用户端；用户端用Kyber私钥获得原文的预主密钥。最后生成会话密钥（即上文中的共享密钥）的必要条件是 自己一侧的ECDH(E)私钥 + 对方的ECDH(E)公钥 + 预主密钥，由于预主密钥的传输过程受后量子密码的保护，所以这种密钥交换机制可以认为是安全的。除了这种机制的密钥交换以外还有两种Hybrid Post-Quantum Key Exchange，可参考A Comprehensive Survey on Post-Quantum TLS (iacr.org)

    实际上，TLS1.3也可以完全仅由PQC实现，如下图(b)。客户端在ClientHello时发送生成的后量子算法的公钥，服务器随机生成一个序列并由这个公钥生成共享密钥以及其加密，加密结果返回用户端并由私钥解密。这样双方也完成了共享密钥的建立。TLS1.3上的纯PQC的密钥交换的问题在于PQC自身的安全性未经过实践检验，主流仍希望暂时保留经过时间检验认定至少在量子时代到来前安全的传统算法，即Hybrid的模式。
